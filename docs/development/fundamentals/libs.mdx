# Plugin Libraries

argsh supports plugin libraries that extend its functionality. Libraries are installed per-project and resolved via the `import` system.

## Quick start

```bash
# Add a library to your project
argsh lib add data

# Use it in your scripts
import data

local domain version
data::get config.yaml \
  domain=.spec.cluster.domain \
  version=.spec.kubernetes.version
```

## Commands

| Command | Description |
|---------|-------------|
| `argsh lib add <name>` | Install a library |
| `argsh lib list` | List installed libraries |
| `argsh lib remove <name>` | Remove a library |
| `argsh lib install` | Install all from `.argsh.yaml` |

## Project configuration

### `.argsh.yaml`

Declare your project's library dependencies:

```yaml
defaults:
  path_libs: .argsh/libs              # where libs are installed

registries:
  # argsh registry is built-in (ghcr.io/arg-sh/libs)
  myco:
    endpoint: harbor.mycompany.com/argsh

libs:
  argsh@data: ^0.1.0
  argsh@kubernetes: ~0.3.0
  myco@k8s-utils: ^0.2.0
```

### Import resolution

When you `import data`, argsh looks in this order:

1. **Relative to caller** — standard import path (`${ARGSH_SOURCE}/../data`)
2. **Project libs** — `.argsh/libs/data/` (fallback for plain imports)
3. **Bundled** — argsh core libraries (`libraries/`)

## Available libraries

| Library | Description | Registry |
|---------|-------------|----------|
| `data` | YAML/JSON structured data access | `argsh` |

## Creating a library

A library is a directory with:

```text
mylib/
├── argsh-plugin.yml    # metadata
├── mylib.sh            # bash library
└── mylib.bats          # tests
```

```yaml
# argsh-plugin.yml
name: mylib
version: 0.1.0
description: My custom library
requires:
  argsh: ">=0.9.0"
```

See the [libs repository](https://github.com/arg-sh/libs) for examples.
