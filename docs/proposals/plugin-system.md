# Proposal: Plugin System for argsh

## Problem

argsh libraries are bundled in the core repo. Adding new libraries (like `data`) requires releasing a new argsh version. Users can't share or reuse libraries across projects.

## Design principles

- **Local by default** — libs install into the project, not globally
- **No new tooling** — `import` just works, `argsh` handles management
- **Familiar** — like npm/pip for interpreted languages, not Go modules
- **Minimal footprint** — OCI for distribution, no custom registry

## Usage

```bash
# In a script — just the name (version resolved from .argsh.yaml / .argsh.lock)
import data
import k8s-utils

# Project management
argsh lib add data                                # short name, default registry
argsh lib add ghcr.io/mycompany/internal/utils    # explicit registry
argsh lib add data@0.1.0                          # pinned
argsh lib install                                 # install all from .argsh.lock
argsh lib update                                  # update within .argsh.yaml ranges
argsh lib remove data
argsh lib list
```

## How it works

### Resolution order

`import data` resolves in order:

1. **Project libs** — `.argsh/libs/data/` (local, versioned)
2. **Bundled** — argsh core libraries (`libraries/`)
3. **Auto-fetch** — pull from registry if not found locally (opt-in)

### Project structure

```text
myproject/
├── .argsh.yaml          # project config — human-edited (semver ranges)
├── .argsh.lock          # resolved versions — machine-generated (committed)
├── .argsh/
│   └── libs/            # installed plugins (gitignored, like node_modules/)
│       └── data/
│           ├── data.sh
│           └── data.so  # optional platform-specific builtin
└── script.sh
```

### Config: `.argsh.yaml`

Human-edited. Registries as named providers, libs reference them with `provider@name`.

```yaml
# .argsh.yaml
defaults:
  path_libs: .argsh/libs                   # where libs are installed (default)
  # source: ../..                          # default for # argsh source= directive
  # builtin_path: .bin/argsh.so            # where to install core .so

registries:
  # argsh (ghcr.io/arg-sh/libs) is built-in — no need to declare it
  myco:
    endpoint: harbor.mycompany.com/argsh
    # auth: ~/.docker/config.json           # optional, for private registries

libs:
  argsh@data: ^0.1.0                       # → ghcr.io/arg-sh/libs/data:0.1.x
  argsh@kubernetes: ~0.3.0                 # → ghcr.io/arg-sh/libs/kubernetes:0.3.x
  myco@k8s-utils: ^0.2.0                  # → harbor.mycompany.com/argsh/k8s-utils:0.2.x
```

`defaults` section holds project-level settings. All paths relative to `.argsh.yaml`. Extensible for future settings.

`argsh` registry (`ghcr.io/arg-sh/libs`) is always available — no config needed for official libs.
In scripts: `import data` (short, unambiguous) or `import argsh@data` (explicit).

### Lock: `.argsh.lock`

Machine-generated. Exact versions, full OCI refs, checksums. Committed to git.

```yaml
# .argsh.lock
libs:
  argsh@data:
    ref: ghcr.io/arg-sh/libs/data:0.1.2
    digest: sha256:abc123...
  argsh@kubernetes:
    ref: ghcr.io/arg-sh/libs/kubernetes:0.3.5
    digest: sha256:def456...
  myco@k8s-utils:
    ref: harbor.mycompany.com/argsh/k8s-utils:0.2.3
    digest: sha256:789fed...
```

### `.gitignore`

```text
# argsh plugin cache (regenerated by argsh lib install)
.argsh/libs/
```

`argsh lib install` reads `argsh.lock` and populates `.argsh/libs/`.

### Plugin artifact (published)

```text
data-0.1.0/
├── data.sh                  # bash library
├── data-linux-amd64.so      # optional Rust builtin (per-platform)
├── data-linux-arm64.so
├── data-linux-musl-amd64.so
├── data-linux-musl-arm64.so
├── argsh-plugin.yml         # metadata
└── data.bats                # tests
```

```yaml
# argsh-plugin.yml
name: data
version: 0.1.0
description: Structured data access for YAML/JSON
requires:
  argsh: ">=0.9.0"
builtin: true               # has .so files
```

### Libs repo structure (`github.com/arg-sh/libs`)

Monorepo with Cargo workspace. Shared Rust infrastructure, independent lib versions.

```text
github.com/arg-sh/libs/
├── Cargo.toml                    # workspace — shared deps compiled once
├── Dockerfile                    # one build for all .so's
├── shared/                       # shell FFI bridge, common types
│   ├── Cargo.toml
│   └── src/
│       ├── lib.rs
│       └── shell.rs              # reused from argsh core
├── data/                         # has Rust builtin
│   ├── argsh-plugin.yml
│   ├── data.sh
│   ├── data.bats
│   ├── Cargo.toml                # [lib] crate-type = ["cdylib"]
│   └── src/lib.rs                # data::get, data::set, data::each
├── http/                         # has Rust builtin
│   ├── argsh-plugin.yml
│   ├── http.sh
│   ├── http.bats
│   ├── Cargo.toml
│   └── src/lib.rs                # http::get, http::post
├── kubernetes/                   # pure bash — no Cargo.toml
│   ├── argsh-plugin.yml
│   ├── kubernetes.sh
│   ├── kubectl.sh
│   ├── helm.sh
│   └── kubernetes.bats
├── template/                     # pure bash
│   ├── argsh-plugin.yml
│   ├── template.sh
│   └── template.bats
└── .github/
    └── workflows/
        └── publish.yaml          # test all, build .so's, publish each as OCI
```

```toml
# Cargo.toml (workspace root)
[workspace]
members = ["shared", "data", "http"]

[workspace.dependencies]
serde = { version = "1", features = ["derive"] }
serde_json = "1"
serde_yaml = "0.9"
bash-builtins = "0.4"
libc = "0.2"
```

**Core vs plugins:**

- `argsh builtin install` — installs core `libargsh.so` from `github.com/arg-sh/argsh`. Provides `:args`, `:usage`, `import`, `is::*`, `to::*`. This is the foundation — without it, everything still works (pure-bash fallback) but slower and no plugin `.so` loading.
- `argsh lib add` — installs plugin libs from the libs repo. Plugin `.sh` files always work. Plugin `.so` files are optional acceleration (loaded via `enable -f` if core builtin is available).

**Why a workspace:**

- Shared deps (`serde`, `bash-builtins`) compiled once, reused across all libs
- One CI pipeline — single Dockerfile, single build matrix
- Shell FFI bridge (`shared/`) written once, not duplicated per lib
- Pure-bash libs (kubernetes, template) have no Cargo.toml — CI skips .so build
- Each lib gets its own `.so` — no monolithic binary
- Independent versioning via git tags: `data/v0.1.0`, `kubernetes/v0.3.0`

### Distribution via OCI

Plain OCI artifacts. Any OCI-compliant registry — no vendor lock-in.

```bash
# Publish (library author)
argsh lib publish                                    # to registry in argsh-plugin.yml
argsh lib publish --registry harbor.myco.com/argsh   # explicit

# Install (user)
argsh lib add data                                   # default registry
argsh lib add harbor.myco.com/team/k8s-utils@0.2.0  # any registry
```

Works with: `ghcr.io`, Harbor, Docker Hub, AWS ECR, GitLab Registry, any OCI-compliant registry.

### Global install (opt-in)

```bash
# Explicitly global — for CLIs/tools, not project libs
argsh lib add --global data
# Installs to ~/.local/share/argsh/libs/
```

## Comparison

| Aspect | argsh libs | npm | pip | Go modules |
|--------|-----------|-----|-----|------------|
| Default scope | project | project | global* | module |
| Config | .argsh.yaml | package.json | requirements.txt | go.mod |
| Lock file | .argsh.lock | package-lock.json | — | go.sum |
| Registry | OCI (ghcr.io) | npmjs.com | pypi.org | proxy.golang.org |
| Install dir | .argsh/libs/ | node_modules/ | site-packages/ | vendor/ |
| Auto-fetch | opt-in | yes | no | yes |
| Compiled parts | optional .so | native addons | C extensions | compiled |

*pip: global by default, `--user` or venv for local

## Dependencies: keep it simple

No dependency resolution. No transitive dependencies. No version conflict algorithms.

### v1 rules

- **Flat** — all plugins declared in the project's `.argsh.yaml`, no nesting
- **Self-contained** — plugins bundle everything they need, no transitive deps
- **Exact versions in lock** — `.argsh.lock` has exact versions + checksums, no ranges to resolve
- **Conflicts = error** — if two things need incompatible versions, the user fixes it

### What this means

```yaml
# .argsh.yaml — the user manages all deps, no magic
libs:
  argsh@data: ^0.1.0              # argsh registry is built-in
  argsh@kubernetes: ^0.3.0
  # no "data depends on string" — if data needs string, user adds string
```

A plugin's `argsh-plugin.yml` can declare `requires.argsh: ">=0.8.0"` (minimum argsh version) but NOT other plugins. This keeps the resolver trivial: for each lib, find the latest version matching the range, done.

### Future extensibility

The format allows adding optional fields later without breaking v1:

```yaml
# Future argsh-plugin.yml (v2, not v1)
name: kubernetes
version: 0.3.0
requires:
  argsh: ">=0.9.0"
  libs:                    # v2: optional peer dependencies
    data: ">=0.1.0"       # "I work best with data, please also install"
```

v1 ignores `requires.libs`. v2 could warn ("kubernetes recommends data") or error. But v1 plugins are always self-contained — no resolver needed.
