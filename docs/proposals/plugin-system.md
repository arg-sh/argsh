# Proposal: Plugin System for argsh

## Problem

argsh libraries are bundled in the core repo. Adding new libraries (like `data`) requires releasing a new argsh version. Users can't share or reuse libraries across projects.

## Design principles

- **Local by default** вЂ” libs install into the project, not globally
- **No new tooling** вЂ” `import` just works, `argsh` handles management
- **Familiar** вЂ” like npm/pip for interpreted languages, not Go modules
- **Minimal footprint** вЂ” OCI for distribution, no custom registry

## Usage

```bash
# In a script вЂ” just the name (version resolved from .argsh.yaml / .argsh.lock)
import data
import k8s-utils

# Project management
argsh lib add data                                # short name, default registry
argsh lib add ghcr.io/mycompany/internal/utils    # explicit registry
argsh lib add data@0.1.0                          # pinned
argsh lib install                                 # install all from .argsh.lock
argsh lib update                                  # update within .argsh.yaml ranges
argsh lib remove data
argsh lib list
```

## How it works

### Resolution order

`import data` resolves in order:

1. **Project libs** вЂ” `.argsh/libs/data/` (local, versioned)
2. **Bundled** вЂ” argsh core libraries (`libraries/`)
3. **Auto-fetch** вЂ” pull from registry if not found locally (opt-in)

### Project structure

```text
myproject/
в”њв”Ђв”Ђ .argsh.yaml          # project config вЂ” human-edited (semver ranges)
в”њв”Ђв”Ђ .argsh.lock          # resolved versions вЂ” machine-generated (committed)
в”њв”Ђв”Ђ .argsh/
в”‚   в””в”Ђв”Ђ libs/            # installed plugins (gitignored, like node_modules/)
в”‚       в””в”Ђв”Ђ data/
в”‚           в”њв”Ђв”Ђ data.sh
в”‚           в””в”Ђв”Ђ data.so  # optional platform-specific builtin

```

### Config: `.argsh.yaml`

Human-edited. Registries as named providers, libs reference them with `provider@name`.

```yaml
# .argsh.yaml
defaults:
  path_libs: .argsh/libs                   # where libs are installed (default)
  # source: ../..                          # default for # argsh source= directive
  # builtin_path: .bin/argsh.so            # where to install core .so

registries:
  # argsh (ghcr.io/arg-sh/libs) is built-in вЂ” no need to declare it
  myco:
    endpoint: harbor.mycompany.com/argsh
    # auth: ~/.docker/config.json           # optional, for private registries

libs:
  argsh@data: ^0.1.0                       # в†’ ghcr.io/arg-sh/libs/data:0.1.x
  argsh@kubernetes: ~0.3.0                 # в†’ ghcr.io/arg-sh/libs/kubernetes:0.3.x
  myco@k8s-utils: ^0.2.0                  # в†’ harbor.mycompany.com/argsh/k8s-utils:0.2.x
```

`defaults` section holds project-level settings. All paths relative to `.argsh.yaml`. Extensible for future settings.

`argsh` registry (`ghcr.io/arg-sh/libs`) is always available вЂ” no config needed for official libs.
In scripts: `import data` (short, unambiguous) or `import argsh@data` (explicit).

### Lock: `.argsh.lock`

Machine-generated. Exact versions, full OCI refs, checksums. Committed to git.

```yaml
# .argsh.lock
libs:
  argsh@data:
    ref: ghcr.io/arg-sh/libs/data:0.1.2
    digest: sha256:abc123...
  argsh@kubernetes:
    ref: ghcr.io/arg-sh/libs/kubernetes:0.3.5
    digest: sha256:def456...
  myco@k8s-utils:
    ref: harbor.mycompany.com/argsh/k8s-utils:0.2.3
    digest: sha256:789fed...
```

### `.gitignore`

```text
# argsh plugin cache (regenerated by argsh lib install)
.argsh/libs/
```

`argsh lib install` reads `argsh.lock` and populates `.argsh/libs/`.

**Core vs plugins:**

- `argsh builtin install` вЂ” installs core `libargsh.so` from `github.com/arg-sh/argsh`. Provides `:args`, `:usage`, `import`, `is::*`, `to::*`. This is the foundation вЂ” without it, everything still works (pure-bash fallback) but slower and no plugin `.so` loading.
- `argsh lib add` вЂ” installs plugin libs from the libs repo. Plugin `.sh` files always work. Plugin `.so` files are optional acceleration (loaded via `enable -f` if core builtin is available).

**Why a workspace:**

- Shared deps (`serde`, `bash-builtins`) compiled once, reused across all libs
- One CI pipeline вЂ” single Dockerfile, single build matrix
- Shell FFI bridge (`shared/`) written once, not duplicated per lib
- Pure-bash libs (kubernetes, template) have no Cargo.toml вЂ” CI skips .so build
- Each lib gets its own `.so` вЂ” no monolithic binary
- Independent versioning via git tags: `data/v0.1.0`, `kubernetes/v0.3.0`

### Distribution via OCI

Each file is a separate OCI layer with a media type. Standard OCI artifact layout.

```text
ghcr.io/arg-sh/libs/data:0.1.0
в”њв”Ђв”Ђ manifest.json
в”‚   в””в”Ђв”Ђ layers:
в”‚       в”њв”Ђв”Ђ sha256:abc... в†’ data.sh             (application/vnd.argsh.lib.v1+bash)
в”‚       в”њв”Ђв”Ђ sha256:def... в†’ argsh-plugin.yml    (application/vnd.argsh.plugin.v1+yaml)
в”‚       в””в”Ђв”Ђ sha256:ghi... в†’ data-linux-amd64.so (application/vnd.argsh.builtin.v1+so)
в””в”Ђв”Ђ config.json (empty, required by OCI spec)
```

Benefits: selective pull (skip .so if not needed), layer dedup across versions, inspectable with standard OCI tools.

Multi-arch `.so`: platform in filename (`data-linux-amd64.so`), client picks the right one. No manifest list needed for v1.

```bash
# Publish (library author)
argsh lib publish                                    # to registry in argsh-plugin.yml
argsh lib publish --registry harbor.myco.com/argsh   # explicit

# Install (user)
argsh lib add data                                   # default registry
argsh lib add harbor.myco.com/team/k8s-utils@0.2.0  # any registry
```

Works with: `ghcr.io`, Harbor, Docker Hub, AWS ECR, GitLab Registry, any OCI-compliant registry.

### OCI client implementation

Vendored from [ocipkg](https://github.com/termoshtt/ocipkg) (Apache-2.0 + MIT). Only the sync client (~300 lines) using `ureq` вЂ” no tokio, no async runtime. Built into `libargsh.so`.

```rust
// builtin/src/oci.rs вЂ” vendored from ocipkg
// - get_manifest() в†’ pull manifest
// - get_blob() в†’ download layer by digest
// - bearer token auth (ghcr.io, Harbor, ECR)
// - ~300 lines, ureq as only HTTP dependency
```

Bash fallback (no .so): downloads tarball from GitHub releases via `curl`.

### Global install (opt-in)

```bash
# Explicitly global вЂ” for CLIs/tools, not project libs
argsh lib add --global data
# Installs to ~/.local/share/argsh/libs/
```

| Aspect | argsh libs | npm | pip | Go modules |

| Default scope | project | project | global* | module |

## Dependencies: keep it simple

No dependency resolution. No transitive dependencies. No version conflict algorithms.

### v1 rules

- **Flat** вЂ” all plugins declared in the project's `.argsh.yaml`, no nesting
- **Self-contained** вЂ” plugins bundle everything they need, no transitive deps
- **Exact versions in lock** вЂ” `.argsh.lock` has exact versions + checksums, no ranges to resolve
- **Conflicts = error** вЂ” if two things need incompatible versions, the user fixes it

### What this means

```yaml
# .argsh.yaml вЂ” the user manages all deps, no magic
libs:
  argsh@data: ^0.1.0              # argsh registry is built-in
  argsh@kubernetes: ^0.3.0
  # no "data depends on string" вЂ” if data needs string, user adds string
```

A plugin's `argsh-plugin.yml` can declare `requires.argsh: ">=0.8.0"` (minimum argsh version) but NOT other plugins. This keeps the resolver trivial: for each lib, find the latest version matching the range, done.

### Future extensibility

The format allows adding optional fields later without breaking v1:

```yaml
# Future argsh-plugin.yml (v2, not v1)
name: kubernetes
version: 0.3.0
requires:
  argsh: ">=0.9.0"
  libs:                    # v2: optional peer dependencies
    data: ">=0.1.0"       # "I work best with data, please also install"
```

v1 ignores `requires.libs`. v2 could warn ("kubernetes recommends data") or error. But v1 plugins are always self-contained вЂ” no resolver needed.
